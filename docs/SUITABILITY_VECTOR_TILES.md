# Suitability grid — PostGIS vector tiles (MVT)

**Status:** implemented on branch `feature/suitability-vector-tiles`.
**Scope:** how the suitability candidate map is served and rendered. It does **not**
change the suitability scoring methodology, policy, or any stored value — see
[SUITABILITY_POLICY_V1.md](SUITABILITY_POLICY_V1.md) for the analysis itself.

This is a self-hosted PostgreSQL/PostGIS + FastAPI + MapLibre design. There is **no**
Oracle Autonomous Database, Oracle Analytics Cloud, Oracle Spatial Studio, Oracle Map
Visualization, external tile service, or separate tile server involved — tiles are
generated on demand by the existing backend against the existing PostGIS database and
reverse-proxied by the existing Caddy on ports 80/443 only.

---

## 1. Why the `limit=2000` GeoJSON map load was removed

The map previously rendered candidates through `GET /api/v1/suitability/candidates`
with `bbox`, `profile`, and `limit=2000`. That approach had two honesty problems:

- **Partial map.** With ~47,893 candidate cells in the production run, a 2,000-row
  cap meant the map showed only a subset of the matching cells, surfaced to the user
  as *"지도 영역 내 2,000 / 47,609개 표시 (뷰포트 제한)"*. The map implied a complete
  screening when it was showing a slice.
- **Status starvation.** The list endpoint ordered eligible/ranked cells before
  review and excluded cells, so the 2,000-row limit could drop review/excluded cells
  even when their visibility checkboxes were enabled.

The map now serves the **complete** grid as Mapbox Vector Tiles (MVT) generated by
PostGIS. The browser transfers only the tiles the current viewport/zoom needs; there
is no row cap, no `bbox` React state driving a fetch, no 300 ms debounce, and no
`AbortController` for candidate rendering. Every candidate cell of the selected run is
reachable by panning/zooming, and the three statuses never compete for a row budget.

## 2. Endpoint contract

```
GET /api/v1/suitability/tiles/{run_id}/{profile}/{z}/{x}/{y}.mvt
```

Example: `/api/v1/suitability/tiles/47/access_focused/9/436/201.mvt`

| Path param | Type | Validation |
|------------|------|------------|
| `run_id`   | int  | Must exist and be a `SUCCEEDED` run (reuses `_resolve_run_id`); otherwise `404 RUN_NOT_FOUND`. |
| `profile`  | enum | One of `baseline`, `equal`, `equity_focused`, `access_focused`; otherwise `422`. |
| `z`        | int  | `0..22` (Web-Mercator pyramid); otherwise `422`. |
| `x`, `y`   | int  | `0 .. 2^z − 1`; otherwise `422 INVALID_TILE_COORDINATE`. Non-integer/injection segments fail int validation → `422`. |

The `run_id` and `profile` are in the **path** (not query), so each tile URL is
immutable and cacheable — a given `(run, profile, z, x, y)` always yields the same
bytes because a run is never mutated in place.

### Responses

| Case | Status | Body |
|------|--------|------|
| Tile overlapping candidates | `200` | non-empty MVT (`application/vnd.mapbox-vector-tile`) |
| Tile outside the project area (no candidates) | `200` | valid **empty** tile (0 bytes) — never a 5xx |
| `If-None-Match` matches the ETag | `304` | empty, cache headers repeated |
| Unknown / non-succeeded run | `404` | `{"detail": {"error": "RUN_NOT_FOUND", ...}}` |
| Invalid profile / z / x / y | `422` | standard validation error |

Response headers:

```
Content-Type:  application/vnd.mapbox-vector-tile
Cache-Control: public, max-age=31536000, immutable
ETag:          "suit-<run>-<profile>-<z>-<x>-<y>"
```

Caddy passes `Content-Type`, `Cache-Control`, and `ETag` through unchanged (it only
adds the global security headers and transport `encode zstd gzip`; it does not strip
or rewrite the tile content type or cache headers). The candidate **detail** endpoint
is *not* marked immutable — only tiles are.

## 3. Source-layer and feature properties

The tile has a single source-layer:

```
candidates
```

Each feature carries only the lightweight attributes the map renders/inspects with:

`candidate_id`, `candidate_key`, `status`, `rank`, `score`, `provisional_score`,
`zoning_score`, `road_score`, `equity_score`, `demand_score`, `sigungu_region_code`,
`sigungu_region_name`.

Heavy fields are **deliberately excluded** from every tile feature and remain on the
detail endpoint (below): `raw_components`, `component_provenance`,
`nearest_road_provenance`, the full `exclusion_reasons` / `review_reasons` arrays,
`profile_totals` for all profiles, and all other detailed provenance.

### Scoring semantics (identical to the read API)

- **ELIGIBLE** — `score` holds the selected profile's total; `provisional_score` is absent.
- **REVIEW_REQUIRED** — `score` is absent; `provisional_score` holds the selected profile's total.
- **EXCLUDED** — neither `score` nor `provisional_score` is present (no false final score).
- **`rank`** — the selected profile's `profile_ranks->>profile` (absent for review/excluded).

MVT encodes `NULL` attributes by omission, so the absence of `score`/`rank` on a
review/excluded feature is the on-the-wire representation of "no value".

## 4. PostGIS spatial-query strategy

The query (parameterized; every user value is a bound parameter) is:

1. Build the tile envelope in EPSG:3857 with `ST_TileEnvelope(:z, :x, :y)`.
2. Filter candidate geometry in its stored **EPSG:4326** using
   `geometry && ST_Transform(ST_TileEnvelope(:z,:x,:y), 4326)` — a bounding-box overlap
   that uses the existing GiST index `idx_suitability_candidates_geometry`.
3. Transform **only the matched** geometries to EPSG:3857.
4. `ST_AsMVTGeom(..., ST_TileEnvelope(:z,:x,:y), 4096, 64, true)` clips to the tile.
5. `ST_AsMVT(row, 'candidates', 4096, 'geom')` produces the tile.

This is **filter-before-transform**: the index prunes to the tile's cells before any
`ST_Transform`, instead of transforming ~48k rows per tile. No new column or index was
added — the run reuses the GeoAlchemy2-created 4326 GiST index.

### EXPLAIN (representative, ~20k-cell synthetic grid)

- Viewport tile (z12): `Bitmap Index Scan on idx_suitability_candidates_geometry`,
  ~95 features, ~2 KB, ~6 ms. No sequential scan.
- Zoomed-out tile (z9): still a GiST `Bitmap Index Scan`, ~3.7k features, ~70 KB
  (uncompressed; Caddy gzips on the wire). No single tile ships the whole dataset.

The candidate vector source is bounded at `maxzoom: 14` in the client, so higher
interactive zooms overzoom a z14 tile rather than requesting a swarm of sub-cell tiles.

## 5. Frontend rendering

- MapLibre `type: "vector"` source, `tiles: ["…/{z}/{x}/{y}.mvt"]`, `source-layer: "candidates"`.
- The tile URL is built by `suitabilityTileUrl(runId, profile)` from `apiBaseUrl()`
  (empty in production → resolved to `window.location.origin`, because MapLibre fetches
  tiles from a Web Worker whose base is a `blob:` URL and cannot resolve a bare relative
  path). **Same-origin by construction — no host, IP, or domain is hardcoded.**
- Switching profile changes the URL, so the source is removed and re-added (a vector
  source's tiles are immutable once added).
- **Colors:** eligible cells use a stable 5-class score palette with fixed interior
  breaks `[20, 40, 60, 80]` over the 0–100 domain (0–20 · 20–40 · 40–60 · 60–80 ·
  80–100). These are deterministic, **not** per-viewport quantiles, so identical cells
  keep their color as the user pans. Review = amber + dashed outline; excluded = muted.
- **Status filtering** is a MapLibre filter on the complete tile layer
  (`["in", ["get","status"], ["literal", visibleStatuses]]`); excluded is hidden by
  default. Because there is no row limit, no status can be starved.
- **Click → detail:** clicking a tile feature reads `candidate_id` and calls
  `GET /api/v1/suitability/candidates/{candidate_id}?profile=…` for full provenance,
  which populates the detail panel. The selected candidate's full geometry is drawn as
  a separate GeoJSON highlight overlay and the map `fitBounds`/`flyTo`s to it —
  unchanged from before.

## 6. Backward compatibility

`GET /api/v1/suitability/candidates` (GeoJSON list) and
`GET /api/v1/suitability/candidates/{candidate_id}` (detail) are unchanged and still
served; the client still uses the detail endpoint. Only the *map's* rendering source
changed. `/policies`, `/runs`, `/runs/latest`, and `/summary` are untouched.

---

## 7. Deployment validation & rollback

- **No migration.** The endpoint reuses the existing schema and GiST index, so nothing
  is added to Alembic and `rollback-app.sh` can revert the application cleanly (it only
  refuses to roll back *below* the DB's current migration, which is unchanged here).
- **Validate after deploy** (see [DEPLOYMENT.md](DEPLOYMENT.md) and `smoke-test.sh`):
  - `GET /health` → `{"status":"ok","database":"ok"}`.
  - A direct tile fetch returns `200`, `Content-Type: application/vnd.mapbox-vector-tile`,
    `Cache-Control: public, max-age=31536000, immutable`, and a non-empty body over the
    capital region (do not print the binary body).
  - In the browser: network requests use `.mvt` tile URLs, the old
    "2,000 / total (뷰포트 제한)" copy is gone, and status toggles + candidate detail work.
- **Rollback:** `./scripts/deployment/rollback-app.sh` (app image/commit only). The
  Postgres volume is never touched; never `docker compose down -v`. Reverting the app
  restores the previous GeoJSON map behavior with no data change.
